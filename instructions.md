# Руководство разработчика (Project Instructions)

Добро пожаловать в проект! Данный документ описывает архитектурные стандарты, принципы написания кода и правила работы с Unity API в нашем проекте. Пожалуйста, внимательно ознакомьтесь с ними перед началом работы.

## 1. Базовые архитектурные принципы

Наш проект строится на принципах **Чистой архитектуры (Clean Architecture)**, **SOLID** и стремится к минимизации технического долга.

*   **SOLID:**
    *   **SRP:** Один класс — одна ответственность. Разделяйте логику вычислений, управление UI и работу с данными.
    *   **OCP:** Пишите код, открытый для расширения, но закрытый для изменения (используйте интерфейсы, а не `else if`).
    *   **LSP:** Наследники не должны ломать поведение базовых классов. Избегайте `NotImplementedException`.
    *   **ISP:** Создавайте узконаправленные интерфейсы (например, отдельно `IWorkable` и `IEatable`, а не общий `IWorker`).
    *   **DIP:** Модули верхнего уровня (игровая логика) зависят от абстракций (интерфейсов), а не от конкретных реализаций (драйверов БД, конкретного UI).
*   **KISS & DRY:** Не усложняйте без необходимости, но избегайте бездумного дублирования кода. Упрощайте, но не превращайте код в монолитный хаос.
*   **Инкапсуляция:** Скрывайте внутреннее состояние объектов. Используйте свойства (`get` / `private set`), а не публичные поля.

## 2. Глобальная архитектура (Чистая архитектура + Паттерны)

Мы разделяем логику на слои с четкими границами:

### 2.1. Entities (Сущности)
Чистые C#-классы (не `MonoBehaviour`). Содержат бизнес-объекты и независимые правила (например, `Item`, `PlayerStats`).

### 2.2. Interactors (Use Cases / Интеракторы)
Сердце бизнес-логики.
*   Один Interactor отвечает за **один конкретный сценарий** (например, `BuyItemInteractor`).
*   Не знает о Unity API, UI или базах данных. Работает только с интерфейсами (`IRepository`) и сущностями.

### 2.3. Паттерн Repository
Используется для работы с данными.
*   **Интерфейс** (например, `IInventoryRepository`) лежит в слое бизнес-логики.
*   **Реализация** (например, `JsonInventoryRepository` или `SqliteInventoryRepository`) лежит в слое инфраструктуры.
*   **Правило:** Бизнес-логика никогда не вызывает `PlayerPrefs` или `File.WriteAllText` напрямую!

## 3. Архитектура UI: Паттерн MVC

Работа с пользовательским интерфейсом (Canvas) строится строго по паттерну **Model-View-Controller**.

*   **Model (Модель):** Чистый C#-класс. Хранит данные (например, очки, здоровье) и логику их изменения. Вызывает события (C# `event` или `Action`) при изменении данных.
*   **View (Представление):** `MonoBehaviour`-класс. Содержит ссылки на UI-элементы (`[SerializeField] private Text / Slider`). **Не содержит логики вычислений**. Только методы для обновления визуала и проброс кликов кнопок.
*   **Controller (Контроллер):** `MonoBehaviour`-класс (или чистый класс, создаваемый Bootstrapper-ом). Связывает Model и View.
    *   *Поток:* Controller подписывает View на события Model. Controller принимает ввод от View и вызывает методы Model.

## 4. Управление зависимостями (DI) и Сервисы

Мы используем паттерн **Dependency Injection (DI)** (вручную или через фреймворк, например, Zenject).

*   **Никаких синглтонов-спагетти:** Избегайте жестких привязок вроде `AudioService.Instance.Play()`.
*   **Интерфейсы для сервисов:** Каждый сервис должен иметь интерфейс (`IInputService`, `IAudioService`). Логика должна ссылаться на интерфейс, а не на реализацию.
*   **Bootstrapper / Installer:** В проекте должна быть единая точка входа (скрипт на стартовой сцене), которая инициализирует сервисы, связывает их и передает зависимости (подход *Push*).
*   **Service Locator:** Допускается только в исключительных случаях (или если DI-фреймворк избыточен для конкретного микро-модуля), но предпочтение всегда отдается внедрению через конструктор/метод `Construct`.

## 5. Правила работы с MonoBehaviour

Скрипты, наследуемые от `MonoBehaviour`, служат только как "мост" между игровым движком Unity и нашей архитектурой.

1.  **Не используйте MonoBehaviour, если он не нужен:** Сервисы, модели данных и интеракторы должны быть обычными C#-классами.
2.  **Минимизируйте логику в `Update`:**
    *   Не делайте тяжелых вычислений каждый кадр.
    *   Используйте событийно-ориентированный подход (Events/Delegates) вместо постоянного опроса состояния (polling).
3.  **Кэширование компонентов:**
    *   Никогда не вызывайте `GetComponent<T>()` или `FindObjectOfType<T>()` внутри `Update()`.
    *   Получайте ссылки в `Awake()` или `Start()` и сохраняйте их в `private` переменные.
4.  **Разделение логики и физики:**
    *   Ввод и визуальные таймеры обрабатывайте в `Update()`.
    *   Взаимодействие с Rigidbody и физикой — строго в `FixedUpdate()`.
    *   Следование камеры за игроком — в `LateUpdate()`.
5.  **Доступ к полям в инспекторе:**
    *   Используйте `[SerializeField] private Type _name;` вместо `public Type Name;`, чтобы сохранить инкапсуляцию, но иметь доступ из инспектора Unity.

## 6. Работа с Canvas и UI-фреймворком

*   Используйте `RectTransform` (Якоря/Anchors и точки опоры/Pivot) для создания адаптивного интерфейса.
*   Настраивайте `Canvas Scaler` на режим *Scale With Screen Size* с заданным референсным разрешением (например, 1920x1080).
*   Разделяйте UI по Canvas: для HUD (Screen Space - Overlay) и для элементов в мире (World Space).
*   Следите за компонентом `Graphic Raycaster`: удаляйте его с Canvas, если на нем нет кликабельных элементов, чтобы сэкономить ресурсы.

## 7. Хранение данных

Выбор способа хранения зависит от задачи:
*   `PlayerPrefs`: **Только** для простых настроек (громкость, качество графики). Не используйте для хранения инвентаря или прогресса.
*   **JSON (`System.IO`)**: Для сохранения состояния игры, профилей игроков. Сериализуйте чистые классы данных (Data Transfer Objects), а не модели с бизнес-логикой.
*   **SQLite**: Для проектов со сложными выборками, каталогами предметов или большими объемами локальных данных.

## 8. Работа с системой ввода (New Input System)

В нашем проекте используется пакет **Unity.InputSystem**. Весь пользовательский ввод должен считываться строго из настроенного ассета `InputSystem_Actions` (Action Maps: Player, UI и т.д.). 

Мы не используем старый класс `UnityEngine.Input` (например, `Input.GetKeyDown`).

### 8.1. Базовый пример работы с ассетом
Ниже приведен пример того, как технически осуществляется подписка на действия из ассета `InputSystem_Actions` и чтение значений (например, вектора движения). 

```csharp
using UnityEngine;
using UnityEngine.InputSystem;

namespace forces
{
    public class CameraSideController : MonoBehaviour
    {
        [Header("Input Settings")]
        [SerializeField] private InputActionAsset _inputAsset;
        private InputAction _moveAction;
        
        // Флаг, чтобы предотвратить множественные переключения при зажатии
        private bool _isAxisInUse = false; 

        private void Awake()
        {
            if (_inputAsset != null)
            {
                // Ищем Action Map "Player", как задано в Input Actions Editor
                var playerMap = _inputAsset.FindActionMap("Player");
                if (playerMap != null)
                {
                    // Ищем конкретное действие "Move"
                    _moveAction = playerMap.FindAction("Move");
                }
            }
        }

        // Обязательно включаем и выключаем Action при активации/деактивации объекта!
        private void OnEnable()
        {
            if (_moveAction != null) _moveAction.Enable();
        }

        private void OnDisable()
        {
            if (_moveAction != null) _moveAction.Disable();
        }

        private void Update()
        {
            HandleInput();
        }

        private void HandleInput()
        {
            if (_moveAction == null) return;

            // Читаем значение оси X (влево/вправо)
            // A = -1, D = 1
            float moveX = _moveAction.ReadValue<Vector2>().x;

            // Логика "Нажатия": срабатывает один раз при отклонении стика/нажатии кнопки
            if (Mathf.Abs(moveX) > 0.5f)
            {
                if (!_isAxisInUse)
                {
                    if (moveX > 0) Debug.Log("D (Вправо)");  
                    else Debug.Log("A (Влево)");           
                    
                    _isAxisInUse = true; // Блокируем, пока кнопка не будет отпущена
                }
            }
            else
            {
                // Сбрасываем флаг, когда стик/кнопка отпущены (близки к 0)
                _isAxisInUse = false;
            }
        }
    }
}
```

### 8.2. Интеграция в архитектуру (Паттерн Service)
Согласно **пункту 4 (Управление зависимостями)** нашего руководства, игровая логика не должна зависеть от конкретной реализации ввода напрямую в каждом скрипте. 

Приведенный выше код — это отличная техническая база, но в рамках архитектуры проекта этот код должен быть **инкапсулирован внутри сервиса ввода**.

**Правильный флоу в проекте:**
1. У нас есть интерфейс `IInputService` (в слое бизнес-логики):
   ```csharp
   public interface IInputService
   {
       Vector2 GetMoveDirection();
       bool IsInteractPressed();
   }
   ```
2. Мы создаем класс-адаптер `NewInputSystemAdapter : IInputService`, который внутри себя реализует логику поиска `InputActionAsset`, `Enable/Disable` и `ReadValue` (как в примере выше).
3. Игровые контроллеры (например, контроллер перемещения персонажа) запрашивают этот сервис через DI (Dependency Injection) или Service Locator и просто вызывают `_inputService.GetMoveDirection()`, ничего не зная о самом ассете `InputSystem_Actions`.
